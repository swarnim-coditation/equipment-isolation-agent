Equipment Isolation Agent Blueprint

This document defines the proposed CNVRT Equipment Isolation Agent flow, request and response contracts, data dependencies, validation rules, and implementation considerations.

Primary Objective Isolation planning
Source Of Truth Unigraph + CNVRT
Consumer CNVRT UI

Use Case

CNVRT users inspect P&IDs, equipment, instruments, and plant graph data. The target interaction is a context-menu action where the user selects an equipment item, instrument, or line and chooses Isolate.

CNVRT invokes the Equipment Isolation Agent with the selected entity context. The agent identifies isolation actions using Unigraph connectivity and CNVRT visual data. CNVRT consumes the output to display an isolation report and highlight resolvable isolation points on the P&ID image.

The selected entity, Unigraph unit context, and P&ID job context must refer to the same drawing or unit. The agent must warn or reject the request when this cannot be verified.

End-To-End Flow

1

User Selects Isolate

The user selects an equipment item, instrument, or line in CNVRT and chooses the isolation action from the context menu.

OutputSelected entity context.
2

CNVRT Calls The Agent

CNVRT sends project, collection, job, selected entity, and authentication context to the agent runtime.

RuntimeLangflow prototype or standalone agent API.
3

Agent Validates Context

The agent confirms that the selected entity belongs to the supplied project, collection, and P&ID job.

DataProject, collection, job, and drawing metadata.
4

Agent Traverses Unigraph

The agent uses graph relationships to identify connected nozzles, lines, valves, instruments, and boundary candidates.

DataUnigraph physical connectivity.
5

Agent Selects Isolation Candidates

The agent chooses valid isolation devices by graph path, entity class, service type, and proximity to the selected entity boundary.

OutputRanked deterministic candidates.
6

Agent Builds Evidence Package

The agent gathers graph paths, candidate barriers, P&ID/HILT/STLM visual JSON, image context, instruments, line metadata, and plant instructions into a structured evidence state.

RuleMissing safety devices remain missing evidence.
7

LLM Requests More Evidence

The LLM may request approved Gremlin/API/image tools to look for blinds, bleeds, vents, drains, pressure indicators, bypasses, and unresolved source paths.

GuardrailNo raw component invention or arbitrary production Gremlin.
8

Validator Checks Isolation Assurance

A deterministic validator decides whether the plan is positive, proven, unproven, not isolated, or insufficiently evidenced.

RuleThe LLM cannot override assurance status.
9

Agent Resolves Visual Coordinates

The agent uses HILT and related CNVRT visual data to resolve image bboxes for candidates.

RuleNo bbox is returned unless it is safely resolved.
10

CNVRT Displays The Result

CNVRT renders isolation report rows, assurance status, ordered procedure steps, missing evidence, and P&ID markers with valid bboxes.

ConsumerIsolation report, procedure, and P&ID visualizer.

Isolation Policy Configuration

The configuration should define business policy, not encode every graph scenario. Complex graph behavior such as branch handling, loop detection, path search, and fallback traversal belongs in the agent algorithm.

Configurable Policy
  • Maximum traversal depth.
  • Eligible isolation classes.
  • Classes that are excluded from final isolation output.
  • Conditional classes such as check valves and control valves.
  • Whether positive isolation is preferred.
  • Whether line break or blind/spade options should be included.
  • Maximum candidates per equipment boundary or path.
  • Natural-language plant instructions and work-scope flags.
Algorithmic Behavior
  • Traverse all relevant connected nodes from each equipment boundary.
  • Detect branches and evaluate each branch path separately.
  • Continue beyond non-final nodes such as pipes, nozzles, tees, reducers, check valves, and control valves.
  • Stop or warn when another equipment boundary is reached before an isolation point.
  • Detect loops and prevent repeated traversal.
  • Rank valid candidates by path role, distance, depth, and isolation quality.
  • Use P&ID/HILT/STLM evidence when graph classes do not expose blinds, bleeds, drains, vents, or gauges.

Agent Processing Model

The processing model separates entity resolution, context validation, graph traversal, candidate selection, visual placement, and response construction.

Phase 1

Resolve Selected Entity

Inputs
  • selected_entity_type
  • selected_tag
  • selected_entity_id
  • selected_unigraph_node_id
  • selected_hilt_node_id
Logic
  • Prefer stable IDs over text tags.
  • Resolve the selected object to a Unigraph vertex where possible.
  • Record all identifiers for traceability.
Phase 2

Validate Drawing Context

The agent confirms that the selected entity, Unigraph unit, CNVRT collection, and P&ID job refer to the same drawing context.

Phase 4

Select Isolation Candidates

The traversal should be broad, but final candidate selection should be narrow. Pipes, nozzles, tees, reducers, containers, labels, and grouping nodes may be required for path context, but they should not become final isolation points unless policy explicitly allows the class or metadata indicates an isolation function.

Candidate Signals
  • entity_class such as gate_valve, ball_valve, blind_flange.
  • valve_type, fluid_service, service tag, or line tag.
  • Path from selected entity boundary.
  • Traversal depth and source nozzle.
Selection Rules
  • Evaluate each equipment boundary connection as a separate path search.
  • Prefer the first valid isolation device on each path when policy allows.
  • Continue traversal through conditional classes such as check valves or control valves when they are not acceptable final isolation points.
  • Group candidates by source nozzle, connected line, or branch path.
  • Keep candidate ID, source component, traversal depth, and path trace.
  • Do not use an LLM to invent isolation points.
Phase 5

Build Evidence And Run Agentic Planning Loop

The agent packages deterministic evidence before the LLM reasons over it. The LLM can request more evidence through approved tools, but it cannot invent components, IDs, bboxes, or graph paths.

Evidence Inputs
  • Unigraph nozzles, paths, valves, tees, and line groups.
  • P&ID/HILT/STLM visual JSON and image-region evidence.
  • Instrument tags such as PI, TI, and level indicators.
  • Plant instructions and work-scope flags.
Approved Tool Requests
  • Find blinds, spades, flanges, bleeds, vents, drains, and pressure indicators.
  • Find bypasses and alternate routes around selected barriers.
  • Fetch P&ID visual JSON or inspect image regions when graph evidence is incomplete.
  • Stop when the validator reaches a terminal status or the query budget is exhausted.
Phase 6

Validate Isolation Assurance

A deterministic validator decides the final safety status. The LLM may explain the result, but it cannot override the validator.

Assurance Status
  • complete_positive_isolation
  • complete_proven_isolation
  • provisional_unproven_isolation
  • not_isolated
  • insufficient_data
Validation Checks
  • Every boundary path has a valid barrier.
  • No unresolved bypass or alternate route remains.
  • Proven isolation has bleed, vent, drain, gauge, pressure indicator, or test evidence.
  • Positive isolation has confirmed blind, spade, blank flange, disconnection, or equivalent evidence.
Phase 7

Resolve Visual Coordinates

The graph determines what should be isolated. CNVRT visual data determines where each point appears on the P&ID image.

API / SourceData FetchedPurposeFailure Behavior

Phase 8

Build Response Payload

Included Data
  • Project, collection, and job context.
  • Isolation point rows.
  • Assurance status and ordered procedure steps.
  • Traceability: candidate ID, source component, path information.
  • Warnings for missing evidence or unresolved visual matches.
Output Rules
  • Return bbox only when confidently resolved.
  • Return tag_number only when present in source data.
  • Reject off-image boxes.
  • Do not silently mix drawing contexts.

CNVRT UI Consumption

Report Table
  • Reads data[].isolation_points[].
  • Displays job, project, collection, tag, class, method, energy type, and reason.
  • Shows unresolved bbox warnings where applicable.
P&ID Visualizer
  • Fetches job details from /jobs/{job_id}.
  • Fetches image from /uploads/{image_id}.
  • Draws markers only when bbox has four valid numbers.

Validation And Safety Rules

  • The agent must traverse broadly for context but apply the isolation policy before returning final isolation points.
  • Directly connected nodes define the equipment boundary; valid isolation points may appear at depth 1, 2, 3, or later depending on topology.
  • The algorithm must support path-based isolation from each boundary connection rather than only direct-neighbor selection.
  • The agent must not fabricate tag_number.
  • The agent must not fabricate bbox.
  • The agent must reject off-image bboxes.
  • The agent must warn or reject when selected entity context does not match the P&ID job context.
  • Every isolation point must include traceability to source component, candidate ID, and matching method where available.
  • Deterministic graph traversal and validation remain authoritative. LLMs may request approved evidence tools and write procedure text, but they must not invent devices, IDs, paths, tags, or bboxes.
  • Partial results are acceptable when unresolved points are clearly marked and warnings are returned.

Current Prototype Notes

  • The current prototype is implemented as Langflow custom nodes.
  • Langflow is not required for the final architecture. The same contract can be implemented by a standalone agent service.
  • The current Langflow output component stores results through /api/v1/flows/end_flow/{flow_id}.
  • A production CNVRT integration should use a stable agent endpoint and a CNVRT-native result storage mechanism.
